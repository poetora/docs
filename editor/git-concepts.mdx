Git 是一个版本控制系统，允许你跟踪文档的更改并与团队成员协作。使用 Git，对于项目中的每个文件，你都可以看到什么发生了变化、何时变化以及为什么变化。Git 还可以轻松恢复到文件的先前版本，如果你需要的话。

Web 编辑器在后台执行一些 Git 操作。了解这些概念将帮助你更有效地使用 Web 编辑器，并与在本地开发环境中工作的团队成员协作。

## 核心 Git 概念

<Accordions>
  <Accordion title="仓库 (Repository)">
    存储所有文件及其历史记录的文档源。Web 编辑器连接到你的仓库以访问和修改内容。
  </Accordion>

<Accordion title="提交 (Commit)">
  特定时间点更改的保存快照。每个提交都包含一条描述发生了什么变化的消息，并在项目历史中创建永久记录。
</Accordion>

<Accordion title="分支 (Branch)">
  分支指向仓库中的特定提交。你的实时文档是从部署分支构建的。你可以拥有任意数量的其他分支，其中包含尚未发布到实时文档的更改。如果你想将分支的更改合并到实时文档中，可以通过拉取请求将分支合并到部署分支中。
  使用分支可以在不影响实时文档的情况下工作，安全地试验新功能，并在发布前获得审查。
</Accordion>

<Accordion title="部署分支 (Deployment branch)">
  项目的主要分支，实时文档内容从此分支构建。对此分支的更改会自动发布到你的文档站点。通常称为
  `main`，但你可以将任何分支设置为部署分支。
</Accordion>

<Accordion title="拉取请求 (Pull request)">
  一种提议将分支上的更改合并到实时文档中的方式。允许在更改上线前进行审查和讨论。通常称为
  PR，在 GitLab 中也称为合并请求。
</Accordion>

  <Accordion title="差异 (Diff)">
    差异显示文件两个版本之间的更改。在审查拉取请求时，差异突出显示已添加、删除或修改的内容，使识别更改变得容易。
  </Accordion>
</Accordions>

## Web 编辑器如何使用 Git

Web 编辑器通过 [GitHub 应用](/settings/github) 或 [GitLab 集成](/settings/gitlab) 连接到你的 Git 仓库并自动化常见的 Git 操作。

当你：

- **打开文件**：编辑器从你的仓库获取最新版本，确保你始终使用最新内容工作。
- **进行更改**：编辑器将你的更改跟踪为草稿，当你准备保存工作时可以成为提交。
- **保存更改**：编辑器对你的更改进行提交，在项目历史中保存你的工作。
- **创建分支**：编辑器在你的仓库中创建新分支，具有仓库访问权限的任何人都可以使用该分支进行协作和审查更改。
- **在部署分支上发布**：编辑器直接提交并推送到你的部署分支，立即发布你的更改。
- **在其他分支上发布**：编辑器创建拉取请求，允许你在将更改合并到部署分支之前从他人那里获得反馈。

## Git 最佳实践

每个团队都会发展自己的工作流程和偏好，但这些是一些帮助你入门的一般最佳实践。

- **编写描述性提交消息**：使用主动语言具体说明发生了什么变化。
- **使用描述性分支名称**：分支名称应该解释正在进行的工作，并对查看仓库中分支的人有意义。
- **保持分支专注**：将分支上的更改专注于特定任务或项目。
- **合并后删除分支**：当不再需要分支时删除它们，以保持仓库整洁。
