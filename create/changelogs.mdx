## 设置更新日志

<Steps>
  <Step title="为更新日志创建页面">
    1. 在文档中创建一个新页面，例如 `changelog.mdx` 或 `updates.mdx`。
    2. 将更新日志页面添加到 `docs.json` 的导航配置中。
  </Step>

  <Step title="在更新日志中添加 Update 组件">
    为每条更新日志记录添加一个 `Update`。

    包含相关信息，例如功能发布、错误修复或其他公告。
  </Step>
</Steps>

## 基本用法

使用 `<Update>` 组件创建一个更新条目：

```mdx Update 基本示例
<Update label="2024-12-17" description="v2.1.0">

## 新功能

- AI 智能写作助手
- 实时协作功能
- 增强的组件库

## 功能改进

- 提升编辑器响应速度 40%
- 优化移动端用户体验

## 问题修复

- 修复浏览器兼容性问题
- 解决大文档加载问题

</Update>
```

## 自定义更新日志

控制用户浏览更新日志的方式，并让他们及时掌握产品动态。

每个 `Update` 的 `label` 属性都会在右侧侧边栏的目录中自动创建一个条目。这是你的更新日志的默认导航方式。

## 使用建议

### 变更日志结构

建议按照以下结构组织变更日志：

```mdx 推荐的变更日志结构
<Update label="YYYY-MM-DD" description="版本号" tags={["类型"]}>
  ## 简短描述

### 新功能 (New Features)

- 功能描述

### 功能改进 (Improvements)

- 改进描述

### 问题修复 (Bug Fixes)

- 修复描述

### 破坏性变更 (Breaking Changes)

- 变更描述和迁移指南

### 弃用通知 (Deprecations)

- 弃用功能和替代方案

</Update>
```

### 标签分类建议

常用的标签分类：

- **`Feature`** - 新功能
- **`Improvement`** - 功能改进
- **`Bug Fix`** - 问题修复
- **`Performance`** - 性能优化
- **`Security`** - 安全修复
- **`UI/UX`** - 界面和体验改进
- **`API`** - API 相关变更
- **`Documentation`** - 文档更新
- **`Major Release`** - 重大版本发布
- **`Hotfix`** - 紧急修复

### 内容编写指南

1. **清晰的标题** - 使用描述性的标题总结主要变更
2. **结构化内容** - 使用标题和列表组织内容
3. **用户视角** - 从用户角度描述变更的影响和价值
4. **包含截图** - 对于 UI 变更，提供前后对比截图
5. **迁移指南** - 对于破坏性变更，提供详细的迁移步骤

更新日志组件是维护产品发布记录和与用户沟通的重要工具。通过一致的格式和清晰的内容，可以帮助用户了解产品的演进历程和最新功能。
